# 出口 IP 质量自动接管设计

## 目标

优化“出口 IP → 出口 IP 质量”主卡片。当部分 IP 情报源打不开、超时、返回错误或只返回残缺字段时，系统应自动选择其他来源中质量更高的有效结果，不让低质量结果长期占据主卡片，也不因单个来源失败阻塞自动检测。

## 检测模型

- 所有出口 IP 来源继续并发请求，避免固定顺序串行重试增加等待时间。
- 每个来源独立超时、失败和标准化，单个来源失败不影响其他来源。
- 首个达到“可判定”门槛的结果立即展示，降低首屏等待时间。
- 后续结果质量更高时自动升级主结果；质量相同或更低时不覆盖当前结果。
- 全部来源完成后重新择优，得到稳定的最终结果。

## 可判定门槛

主卡片的正式结果必须同时包含：

- 格式合法的 IPv4 或 IPv6 地址。
- 可标准化的国家或地区代码。

只有 IP、没有地区的结果属于临时候选：它可以证明出口 IP 存在，但不能完成机房、地区与风险质量判断，也不能阻止后续完整结果接管。

只有当全部来源完成且仍没有达到门槛的结果时，才允许用“仅 IP”候选作为降级展示；如果连合法 IP 都没有，则显示“无法读取出口 IP”。

## 质量评分

候选结果按以下字段计算完整度：

| 字段 | 分值 |
|---|---:|
| 合法 IP | 必需条件 |
| 国家/地区代码 | 40 |
| 国家/地区名称 | 10 |
| ASN | 20 |
| 组织/ISP | 20 |
| 城市/区域 | 10 |

评分更高的候选优先。评分相同时按既有来源优先级决定，保证结果稳定，不根据网络完成顺序随机抖动。

`—`、`未知`、`Unknown`、空字符串等占位值不计为有效字段。

## IPv4 与 IPv6

- IPv4 和 IPv6 分别维护最佳候选。
- 同一协议、同一 IP 的多个来源可以竞争，保留信息最完整的结果。
- 双栈环境同时展示最佳 IPv4 与最佳 IPv6。
- 主结果仍优先使用 IPv4；仅有 IPv6 时使用 IPv6。
- 后续 IPv4 顶替先返回的 IPv6 时，继续沿用现有的多源互证重跑和 WebRTC 重核逻辑。

## UI 行为

- 首个可判定结果到达后，“出口 IP 质量”立即从检测中切换为结果。
- 更高质量来源到达后，主卡片自动更新地区、ASN、组织和来源标签。
- 自动升级不得让评分节点、风险项或多源互证残留旧结果。
- 不在主卡片增加来源失败列表；详细的来源差异继续由“多源交叉”模块承担。
- 用户点击重测时，所有候选、择优状态和旧请求令牌一起清空，过期响应不得回写。

## 失败与降级

- 部分来源失败：静默由其他有效来源接管。
- 所有完整来源失败，但存在合法 IP：显示该 IP，并明确标记“地区情报未完整测出”，按现有未测出规则处理。
- 所有来源均失败：显示“无法读取出口 IP”，按现有未测出规则扣分。
- 任一阶段不把“未测出”解释为可信。

## 验证要求

- 首选来源失败时，下一完整来源接管。
- 最快来源只有 IP 时，不阻止稍后的完整来源接管。
- 较晚但字段更完整的同 IP 来源能够升级主卡片。
- 较晚但质量更差的来源不能覆盖当前结果。
- IPv4、IPv6 分别保留最佳候选并按既有顺序展示。
- 所有完整来源失败但 IP 可用时正确降级。
- 全部来源失败时保持现有未测出行为。
- 重测后的旧响应不能覆盖新一轮结果。
- 完整 E2E、构建与线上首次打开验证通过。
